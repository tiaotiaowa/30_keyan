## 1. 核心结论（先给你最重要的 10 条）

1) **实现层面的可比性风险显著**：Exp2（标准 KD）使用 ChatML + labels 掩码 + 仅对 assistant token 计算 soft/hard loss；但 Exp3–Exp8 多数脚本使用纯文本“问题/答案”拼接，且通常**不构造 labels**，导致 `hard_loss=0`、KL 可能对 prompt/pad 全量计算。

2) **Exp3/Exp4 的“关键动态量”目前是常数**：
- Exp3（可学习权重）在 `exp03_srd/train.py:188` 固定 `idx=0`，关系类型基本由 raw_data[0] 决定，难以体现“按关系类型动态加权”的设计初衷。
- Exp4（CoT）在 `exp04_cot_distill/train.py:185` 固定 `idx=0`，`num_steps` 几乎恒定；且 L_chain 与 L_answer 的 KL 实际相同，只是常数缩放。

3) **Exp7 的 hidden 蒸馏实现存在强随机噪声源**：当师生 hidden 维度不匹配时，使用每次调用重新生成的随机矩阵投影（`exp07_attention/train.py:127`），会导致 hidden_loss 波动大、难复现实验解释。

4) **Exp8 渐进式蒸馏“按难度过滤数据”的逻辑实现了，但主训练流程未启用**：脚本实现 `difficulty_filter`（`exp08_progressive/train.py:153-165`），但主流程未传入过滤参数；实际渐进仅体现在“按 epoch 切换 temperature/alpha”。

5) **Exp3–Exp9 多个 config 的数据路径与 Exp1/2 不一致，存在“相对路径失效”风险**：
- Exp1/2 使用 `../../data/...`（与 `os.chdir(exp_dir)` 兼容）
- Exp3a/3/4/5/6/7/8/9 多数使用 `data/...`（若未在 exp 目录下复制 data/ 子树，将直接找不到数据）

6) **GLM‑5 评测脚本默认 main() 跑的是“模拟预测”**：`evaluate_glm5.py` 若不传 `predictions` 会生成 mock（70%正确），因此直接运行脚本得到的分数**不代表任何模型**。

7) **GLM‑5 采样策略当前是按“dimension 等量抽样”**（`evaluate_glm5.py:392-412`），不是按题目真实分布/难度×关系类型分层；会改变总体估计。

8) **GeoSR‑Bench 当前可用版本是 1000 题（v1.0），与既定 3000 题目标不一致**：需要明确“论文口径到底用 1000 还是 3000”，并保证与数据 10k 分层策略一致。

9) **训练数据总量目前是 6023（geosr_chain/final），与目标 10k 不一致**：后续若按 10k 冻结，需要同步更新训练/评测数据快照与 stats。

10) 结论导向：若不先解决 1/2/5/6 这类“实现口径漂移”，即便高预算跑完 5 次重复，也很难把增益可信地归因到 C1–C6 组件。

## 3. Exp1–Exp9：逐实验“实现细节考量 + 偏差审计 + 研究解释策略”

> 说明：你要求“严格沿用现配”，所以这里把“应该怎么改”写成**研究解释与风险对策**；不直接给代码改动。

### Exp1（B1 Direct‑SFT，对照组）
- 配置：`exp01_direct_sft/config.yaml`
  - `training.max_length=1024`，batch=4，grad_acc=8，lr=2e-4
  - `distillation.enabled=false`
- 关键实现：
  - ChatML 模板在 config 中定义（system_prompt）
  - 评测脚本 `exp01_direct_sft/evaluate.py` 的 RA=启发式语义匹配（非“RA四项加权”）
- 研究解释：
  - 用于衡量“无蒸馏/仅 SFT”的上限与格式遵循能力

### Exp2（B2 Standard‑KD，Forward KL 基线）
- 配置：`exp02_standard_kd/config.yaml`
  - `max_length=1024`，batch=2，grad_acc=16，lr=1e-4
  - `distillation.method=standard_kd`，T=2.0，α=0.5
- 实现亮点（对后续实验是“口径标杆”）：
  - **labels 掩码**：user 部分置 -100（`exp02_standard_kd/train.py:216-229`）
  - **soft loss mask**：只对 `labels!=-100` 的 token 算 KL（`exp02_standard_kd/train.py:278-293`）
  - 对照意义：这里定义了“对话蒸馏的正确算分边界”。
- 研究解释：
  - 若 Exp3–Exp9 未实现同样的 mask 口径，则其增益/退化不可与 Exp2 直接对比。

### Exp3a（B2+C1 Uniform，等权重空间关系蒸馏）
- 配置：`exp03a_uniform_srd/config.yaml`
  - `training.batch_size=8`，grad_acc=16，`data.max_length=2048`
  - `distillation.method=spatial_relation_uniform`，权重全 1.0
- 关键偏差点（会影响“等权重基线”的意义）：
  - 关系类型提取使用 `inputs["input_ids"].argmax().item()` 做 idx（`exp03a_uniform_srd/train.py:173`），与样本索引无关，容易越界回落 composite。
  - 常见结果：C1 形同虚设（几乎总用 composite 权重=1）。
- 研究解释策略：
  - Exp3a 的定位应从“严格均匀加权”退化为“近似普通 KD 但输入模板不同”。

### Exp3（B2+C1 Learnable，可学习权重）
- 配置：`exp03_srd/config.yaml`
  - 初始权重 directional=1.5、topological=1.3、metric=1.0、composite=1.8，`weight_lr=0.01`
- 关键偏差点（直接影响“可学习权重是否真的学到东西”）：
  - `idx=0` 固定（`exp03_srd/train.py:188`）→ relation_type 几乎固定由 raw_data[0] 决定。
  - 权重优化器每 training_step 都 step（`exp03_srd/train.py:217-225`），与 grad_acc 语义可能不同步。
  - KL 在 `reduction=batchmean` 后才乘 weight（`exp03_srd/train.py:195-199`）→ 权重是“全步标量”，不是 per-sample/per-token。
- 研究解释策略：
  - 论文中若宣称“按关系类型动态加权”，需先证明 relation_type 真随样本变化（当前实现难以支撑）。

### Exp4（B2+C2 CoT 思维链蒸馏）
- 配置：`exp04_cot_distill/config.yaml`（α=0.6，normalize_by_steps=true）
- 关键偏差点：
  - L_chain 与 L_answer 都是整体 KL（`exp04_cot_distill/train.py:127-137`），并未按“推理段 token vs 答案段 token”切分。
  - num_steps 来自 raw_data[0]（`exp04_cot_distill/train.py:185-188`）→ normalize 近似常数缩放。
- 研究解释策略：
  - 当前实现更接近“对包含推理文本的整体输出做 KD”，而非“过程监督”。

### Exp5（B2+C3 Reverse KL）
- 配置：`exp05_reverse_kl/config.yaml` 默认纯 reverse（fw=0, rw=1）
- 实现确认：
  - Reverse KL 实现方向正确（`exp05_reverse_kl/train.py:93-108`）
- 风险点：
  - 若无 labels/hard_loss，训练退化为纯 soft；且 KL 可能覆盖 prompt/pad。
- 研究解释策略：
  - 建议在报告中把 Reverse KL 的效果拆成：
    - “幻觉率/保守性变化”（GLM‑5 accuracy vs completeness 的 trade-off）
    - “生成多样性变化”（辅助统计）

### Exp6（B2+C4 自蒸馏，EMA）
- 配置：`exp06_self_distill/config.yaml`（use_ema=true, ema_decay=0.999, self_weight=0.3/teacher_weight=0.7）
- 实现要点：
  - EMA 通过“参数 shadow 替换”获得 ema_logits（见脚本实现摘要）
  - 总 soft= teacher_KL + self_KL，再与 hard 混合
- 风险点：
  - EMA 容易引入“自我确认偏差”，需用 dev 监控是否更保守（completeness 下降）。

### Exp7（B2+C5 注意力/隐藏态蒸馏）
- 配置：`exp07_attention/config.yaml`（attn=0.3, hidden=0.2, logit=0.5）
- 关键偏差点（需要在论文解释里显式说明，否则 reviewers 会质疑）：
  - hidden 投影矩阵每次调用随机生成（`exp07_attention/train.py:127`），导致 hidden_loss 不是稳定的“表征对齐”。
  - 数据加载仍使用 `eval()`（`exp07_attention/train.py:140`），且不构造 labels → hard_loss 可能为 0。
- 研究解释策略：
  - 将 Exp7 结果主要归因为“attention 对齐 + logits KD”，不要过度宣称 hidden 对齐贡献。

### Exp8（B2+C6 渐进式蒸馏）
- 配置：`exp08_progressive/config.yaml`
  - stage（easy/medium/hard）按 epoch 区间切 temperature/alpha
- 实现确认：
  - callback `on_epoch_begin` 切 stage（`exp08_progressive/train.py:44-58`）
  - `difficulty_filter` 虽实现但主流程未启用（`exp08_progressive/train.py:153-165`）
- 研究解释策略：
  - 当前 Exp8 更像“温度/α schedule”，而不是严格的“按难度课程数据”。

### Exp9（Full，GeoKD‑SR 完整方法）
- 配置：`exp09_geo_kd_sr/config.yaml`
  - components loss_weight：C1 0.3 / C2 0.2 / C3 0.2 / C4 0.1 / C5 0.1，C6 为 temperature_multiplier
- 实现要点（必须写进论文方法节，避免被认为是“拼装”）：
  - progressive 只做温度乘数，不切数据
  - attention 蒸馏只用最后一层 attention（而 Exp7 是 all-layers）
  - relation_type 仍固定取 raw_data[0]（见 `exp09_geo_kd_sr/train.py` 片段）
- 研究解释策略：
  - 重点用“组件协同”讲故事，但要同步给出“实现口径限制”，否则 Full 的提升可能来自输入模板/损失边界差异。

## 4. 评测与指标：从“可跑”走向“可发表”的关键设计

### 4.1 自动评测（evaluate.py）的口径与局限
- Exp1/Exp2 的 evaluate.py 计算：
  - RA：启发式（完全匹配/包含/数字/方向词一致）
  - SR‑F1：空间关键词集合匹配
  - BLEU/ROUGE：字符级

**局限**：这套 RA 并非设计文档中的“RA=四项加权评分”，更接近 answer‑match accuracy。

建议定位：
- 把它当成“快速回归指标”（pipeline sanity / 训练是否坏掉）
- 正式论文主指标建议由 GLM‑5/人工 rubric 输出（至少覆盖推理质量/完整性）

### 4.2 GLM‑5 评测系统：已实现的能力 + 需要注意的偏差
- 评分维度：accuracy/completeness/reasoning_quality/fluency（1–5）
- 采样：默认 sample_size=300，按 benchmark 的 `dimension` 做等量分层（`evaluate_glm5.py:392-412`）
- 解析回退：JSON 解析失败 → 四维与 overall 置 3 分（会“拉中值”）
- **关键陷阱**：不提供 predictions 时会生成 mock（70%正确）（`evaluate_glm5.py:497-543`）

建议把 GLM‑5 评测变成“可发表证据”的 3 个措施（设计级，不写代码）：
1) 固定 Prompt 与输出 schema，建立版本号（prompt_v1/prompt_v2）。
2) 记录每题原始响应文本与解析结果；统计“回退比例”。回退率 >1% 即需人工复核。
3) 采样策略改为：按（dimension×difficulty）或按真实分布比例抽样；否则“等量分层”会改变总体估计。

### 4.3 把 GLM‑5 维度映射到 RA（四项加权）
若以 GLM‑5 作为主评：
- 先把 1–5 映射到 0–100（或 0–1）
- 定义：
  - step_score ≈ reasoning_quality
  - answer_score ≈ accuracy
  - completeness_score ≈ completeness
  - clarity_score ≈ fluency
- RA（论文主指标）按设计文档权重计算：
\[
RA = 0.4\cdot step + 0.3\cdot answer + 0.2\cdot completeness + 0.1\cdot clarity
\]

### 4.4 统计检验（高预算版）
- 每个实验 5 次重复：seed 列表建议固定（例如 42/43/44/45/46）
- 主比较集合（建议预注册，避免 p‑hacking 观感）：
  - Exp2 vs Exp1（KD 是否有效）
  - Exp3 vs Exp3a（可学习权重是否优于等权重）
  - Exp4/5/6/7/8 vs Exp2（各组件增益）
  - Exp9 vs Exp2（Full vs 基线）
- 检验：配对 t / Wilcoxon；效应量 Cohen’s d / Cliff’s delta；多重比较 Holm‑Bonferroni。

输出模板（建议）：
- 表A：总体 RA（均值±std，p，effect size）
- 表B：四维分数（accuracy/completeness/reasoning/fluency）
- 表C：按关系类型分桶（directional/topological/metric/composite）

## 5. GeoSR‑Bench：现状与“3k 论文口径”对齐方案

### 5.1 现状
- `geosr_bench_v1.json`：1000题，D1/D2/D3 三维度（`generate_benchmark.py`）
- `benchmark.json`：仅 5 题示例，且字段 schema 与 v1 不同

### 5.2 3k 目标的两条路线（研究设计选择）
A) **扩展生成式 Bench（从 1k → 3k）**：
- 优点：维度覆盖清晰、可控、无需依赖训练数据分布
- 风险：生成题可能与真实数据分布存在 domain gap

B) **从 test 集分层抽样形成 Bench（推荐与你既定“数据优先路线”一致）**：
- 从冻结 test 集按（关系类型×难度）分层抽 3k
- 优点：与训练分布更一致、对比更公平
- 风险：需要先保证 test 的字段完整与泄漏检测

> 当前实现侧 benchmark 的维度是 D1/D2/D3；而你设计文档中的维度/关系类型划分与之不完全一致。建议明确论文最终采用哪种维度体系，并统一写法。

## 6. 外部研究启示（已通过 Exa 联网检索，给出“能用的结论”）

1) **推理蒸馏不要只模仿表面 CoT 文本**：更有效的是对“推理过程的结构/算法性”做迁移（启示：把 Exp4 从“整体 KL + 常数 normalize”推进到“分段/分步监督”）。
2) **课程式蒸馏需要真实的“数据分布渐进”**：只调温度/α 可能不足以获得稳定增益（启示：Exp8 若要讲 curriculum，需要和 difficulty 分层抽样联动）。
3) **LLM‑as‑a‑Judge 评测要显式控制偏差**：固定 prompt、记录解析失败率、建立抽样复核流程，才能成为可发表证据链。
