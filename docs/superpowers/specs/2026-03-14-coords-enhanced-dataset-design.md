# 坐标增强数据集设计方案

> **创建日期**: 2026-03-14
> **需求来源**: 用户请求
> **状态**: 已批准

---

## 一、背景与目标

### 1.1 问题背景
当前 GeoKD-SR 数据集中，`question` 字段只包含实体名称，不包含坐标信息。小模型在训练时需要依赖其内部参数中的地理知识来理解实体位置，这可能导致：
- 模型对不熟悉的地理实体无法正确推理
- 模型的地理知识来源于预训练，而非从数据中学习

### 1.2 目标
创建一个新的数据集变体，在 `question` 中的实体后添加坐标信息，让小模型可以直接从输入文本中学习位置信息。

---

## 二、设计方案

### 2.1 转换逻辑

**输入示例**:
```
question: "渭南市属于陕西省的行政管辖范围之内吗？"
entities: [
  {"name": "陕西省", "type": "province", "coords": [108.9398, 34.3416]},
  {"name": "渭南市", "type": "city", "coords": [109.5099, 34.5024]}
]
```

**输出示例**:
```
question: "渭南市(109.51,34.50)属于陕西省(108.94,34.34)的行政管辖范围之内吗？"
```

### 2.2 坐标格式
- 使用数据中已有的 `entities[].coords` 字段
- 格式：`实体名(经度,纬度)`
- 坐标保留原始精度（不做四舍五入）

### 2.3 字段处理

| 字段 | 处理方式 |
|------|---------|
| `question` | **增强** - 在实体后添加坐标 |
| `answer` | **保持不变** |
| `reasoning_chain` | **保持不变** |
| `entities` | **保持不变** |
| 其他字段 | **保持不变** |

### 2.4 替换策略

使用简单字符串替换：
1. 遍历 `entities` 数组
2. 对每个实体，将 `name` 替换为 `name(coords[0],coords[1])`
3. 按实体名称长度降序排列，先替换长名称（避免"北京"误替换"北京大学"的问题）

### 2.5 数据集范围

| 源文件 | 记录数 | 输出文件 |
|--------|--------|----------|
| `data/final/splits/train.jsonl` | 9,463 | `data/final/split_coords/train.jsonl` |
| `data/final/splits/dev.jsonl` | 1,124 | `data/final/split_coords/dev.jsonl` |
| `data/final/splits/test.jsonl` | 1,183 | `data/final/split_coords/test.jsonl` |

### 4.2 数据示例对比

```json
{
  "question": "从大连看，孝感位于哪个方向？",

  ]
```

```json
{
  "question": "从大连(121.6147,38.914)看，孝感(113.9169,30.9243)位于哪个方向？",
  "entities": [
    {"name": "大连", "coords": [121.6147, 38.914]},
    {"name": "孝感", "coords": [113.9169, 30.9243]}
```

## 五、注意事项

1. **实体名称唯一性**: 同一 question 中的实体名称应该是唯一的
2. **替换顺序**: 按名称长度降序替换，避免部分匹配问题
3. **坐标精度**: 保留原始精度，不做四舍五入
4. **文件编码**: UTF-8
