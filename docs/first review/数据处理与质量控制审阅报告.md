> **审阅日期**: 2026年3月2日
> **审阅版本**: V5.0（数据公平性设计版）
> **审阅人**: 数据处理与质量控制专家

## 一、执行摘要

本次审阅针对GeoKD-SR实验设计方案的数据处理和质量控制部分进行了深度分析。在已有16个问题的基础上，**新发现7个问题**，其中**严重问题2个**、**中等问题3个**、**轻微问题2个**。

### 关键发现

1. **教师模型生成训练数据的循环依赖问题**（严重）：学生模型被用来生成自己的训练数据，存在概念循环
2. **Exp4输入格式不一致问题在已有审阅中已识别**，但未完全解决监督信号不一致的问题
3. **数据生成质量控制标准缺乏可操作性的量化指标**
4. **难度等级判定标准过于主观**，缺乏明确定义
5. **地理实体库规模与数据生成量的匹配度不足**
6. **缺少数据生成过程中的偏差检测机制**
7. **合成数据与原始数据的"等价性"未经验证**

## 二、新发现的问题清单

### 2.1 严重级别问题（必须修改）

#### 🔴 新问题1：教师模型生成训练数据的循环依赖问题

- **位置**: 4.6.1节数据生成流程、C4合成数据蒸馏
- **问题描述**:
  - 方案使用Qwen2.5-7B（教师模型）生成8,000条训练数据
  - 然后用这些数据训练学生模型Qwen2.5-1.5B
  - 再用教师模型蒸馏学生模型
  - 这存在概念循环：教师模型生成的数据质量直接影响学生模型，而学生模型的性能又用来验证蒸馏效果

- **影响分析**:
  - 如果教师模型本身存在空间推理错误，这些错误会被编码到训练数据中
  - 学生模型可能过度拟合教师模型的错误模式，而非学习真正的空间推理能力
  - 蒸馏效果的提升可能仅仅来自于"更多训练数据"，而非蒸馏方法本身

- **具体风险场景**:
  ```
  教师模型错误示例：
  - 问：北京在上海的什么方向？
  - 教师生成数据可能包含：北京在上海的"正西方向"（错误）
  - 学生模型学习这个错误模式
  - 蒸馏过程中，教师模型继续强化这个错误
  ```

- **修改建议**:
  1. **增加独立数据来源**：至少30%的数据来自非教师模型来源（如人工标注、其他LLM生成）
  2. **添加教师模型错误率评估**：在数据生成前，评估教师模型在空间推理任务上的准确率
  3. **分层设计实验**：
     - Exp6a: 使用其他模型（如GPT-4）生成的数据
     - Exp6b: 使用混合数据源（50%教师+50%其他）
     - 对比分析数据来源对蒸馏效果的影响
  4. **在论文中明确声明**数据生成方法的局限性

- **优先级**: P0（必须在数据生成前解决）

#### 🔴 新问题2：Exp6合成数据蒸馏违反消融实验基本假设

- **位置**: 3.1节实验配置表、4.7.1节设计原则、C4组件设计
- **问题描述**:
  - 实验配置表显示Exp6使用"教师生成的数据"
  - 但4.7.1节明确声明"所有实验使用完全相同的数据文件"
  - 这两个要求存在根本矛盾

- **影响分析**:
  - **违反消融实验基本假设**：消融实验要求只有一个变量（C4组件），但Exp6改变了两个变量（方法+数据）
  - **无法分离C4的贡献**：Exp6 vs Exp2的差异可能来自数据差异，而非C4组件
  - **破坏实验公平性**：其他实验（Exp3-5,7-8）使用相同数据，只有Exp6使用不同数据

- **具体对比**:
  ```
  Exp2: Standard-KD + 原始数据(8000条)
  Exp6: Standard-KD + C4 + 合成数据(8000条)

  差异分析：
  - 方法差异：C4组件（预期变量）
  - 数据差异：原始vs合成（混淆变量）
  - 无法确定性能提升来自哪个变量
  ```

- **修改建议**:
  **方案A（推荐）：重新设计C4组件**
  - C4不应改变训练数据，而应改变损失函数或训练策略
  - 例如：C4可以是"自蒸馏损失"，让学生模型在自己的预测上蒸馏
  - 公式：`L_self = KL(P_S(x) || P_S(x_augmented))`，其中x_augmented是数据增强版本

  **方案B：分离数据实验**
  - 新增Exp6-Data: 使用合成数据（无C4组件）
  - 新增Exp6-Method: 使用原始数据+C4损失
  - 通过2×2设计分离数据和方法的贡献

  **方案C：改名为"数据增强实验"**
  - 明确Exp6不是消融实验，而是"数据质量影响实验"
  - 从消融实验序列中移除
  - 在论文中单独讨论数据质量的影响

- **优先级**: P0（必须在最终方案中解决）

### 2.2 中等级别问题（建议修改）

#### 🟡 新问题3：质量控制标准缺乏可操作的量化指标

- **位置**: 4.6.2节质量控制标准表
- **问题描述**:
  - 标准表列出了"≥95%正确"、"≥90%正确"等阈值
  - 但未说明如何自动检测"推理步骤逻辑正确性"
  - 未定义"空间关系类型标注正确性"的判定规则

- **具体问题**:
  ```
  标准声明：
  - "推理链验证": 推理步骤逻辑正确性 ≥90%正确

  实际问题：
  - 如何自动检测"逻辑正确性"？
  - "北京在(116.4°E, 39.9°N)"→"上海在(121.5°E, 31.2°N)"→"计算方位角"
    这三步逻辑是否正确，需要什么验证？
  - 90%的正确率是指每条数据90%的步骤正确，还是90%的数据完全正确？
  ```

- **影响分析**:
  - **质量控制不可执行**：标准无法落实到代码
  - **人工验证量增加**：缺乏自动化导致需要更多人工验证
  - **数据质量不可信**：审稿人会质疑"如何保证质量"

- **修改建议**:
  1. **为每个验证环节设计可执行的检查规则**：
     ```python
     # 推理链逻辑验证示例
     def validate_reasoning_chain(chain, question, answer):
         checks = []
         # 检查1: 推理链长度
         checks.append(len(chain) >= 2)  # 至少2步
         # 检查2: 包含实体坐标
         checks.append(any('坐标' in step or '°' in step for step in chain))
         # 检查3: 包含计算/推理关键词
         checks.append(any(kw in step for step in chain
                           for kw in ['计算', '判断', '因此', '所以']))
         # 检查4: 推理链与答案一致性
         checks.append(extract_direction(chain[-1]) == answer)
         return all(checks)
     ```

  2. **明确定义"正确"的判定规则**：
     - 空间关系验证：通过坐标计算验证方向/距离
     - 实体验证：实体名称在知识库中存在
     - 坐标验证：经纬度在合理范围内（中国：73°E-136°E, 18°N-54°N）

  3. **增加自动化验证覆盖率**：
     - 目标：70%通过自动验证，30%人工验证
     - 自动验证失败的数据进入人工验证队列

  4. **在4.6.2节添加验证规则子表**：
     | 验证项 | 自动规则 | 人工规则 | 覆盖率 |
     |--------|---------|---------|--------|
     | 坐标合理性 | 数值范围检查 | - | 100% |
     | 方向正确性 | 方位角计算 | 抽样10% | 90% |
     | 推理链逻辑 | 关键词+结构检查 | 抽样20% | 70% |

- **优先级**: P1（建议在数据生成前完善）

#### 🟡 新问题4：难度等级判定标准过于主观

- **位置**: 4.5.2节数据分配表、4.2节统一数据格式
- **问题描述**:
  - 数据按难度分配：Easy(30%), Medium(50%), Hard(20%)
  - 但未明确定义"难度"的判定标准
  - "组合推理=最复杂"是主观判断，缺乏实证支持

- **具体问题**:
  ```
  当前定义：
  - 方向关系：基础认知 → Easy
  - 拓扑关系：需要理解概念 → Medium
  - 度量关系：需要数值计算 → Medium
  - 组合推理：多步推理 → Hard

  问题：
  - "北京到上海的距离"（度量）可能比"从北京到上海经过哪些省份"（组合）更难
  - 坐标计算（方向）对普通用户可能比拓扑关系（包含/相邻）更难
  - 难度应该基于用户认知，而非任务复杂度
  ```

- **影响分析**:
  - **数据分布不合理**：可能导致简单任务过多或困难任务不足
  - **训练效率低下**：模型可能在不合适的难度上花费过多时间
  - **评估偏差**：Hard任务如果定义不准，无法真正测试模型上限

- **修改建议**:
  1. **多维度难度定义**：
     ```python
     difficulty_score = (
         0.3 * cognitive_load +      # 认知负荷（需要多少背景知识）
         0.3 * computation_steps +    # 计算步骤（需要多少步推理）
         0.2 * data_requirements +    # 数据需求（需要多少实体/坐标）
         0.2 * ambiguity_level        # 歧义性（答案是否唯一）
     )

     # 分类阈值
     Easy: score < 2.0
     Medium: 2.0 ≤ score < 3.5
     Hard: score ≥ 3.5
     ```

  2. **基于预实验验证难度分类**：
     - 选择100个样本
     - 让多个模型（Qwen-7B, GPT-4, 人类专家）回答
     - 根据准确率反向定义难度

  3. **在4.5.2节添加难度定义表**：
     | 难度 | 认知负荷 | 计算步骤 | 数据需求 | 示例 |
     |------|---------|---------|---------|------|
     | Easy | 常识 | 1-2步 | 2个实体 | 北京在上海的什么方向？ |
     | Medium | 需要概念 | 3-4步 | 2-3个实体 | 从北京到上海经过哪些省份？ |
     | Hard | 需要专业知识 | 5+步 | 3+个实体 | 考虑地形因素，从成都到拉萨的最佳路线？ |

  4. **考虑动态难度调整**：
     - 先按简单规则分类
     - 生成后根据模型表现调整
     - 被所有模型答对的→降级，被所有模型答错的→升级

- **优先级**: P1（建议在数据生成前明确定义）

#### 🟡 新问题5：地理实体库规模与数据生成需求不匹配

- **位置**: 4.6.1节数据生成流程
- **问题描述**:
  - 地理实体库：34个省会+100+地级市+著名山脉/河流+省份/区域 ≈ 200个实体
  - 数据生成需求：8000条数据
  - 按组合计算：200个实体两两组合 = 19,900种可能
  - 但考虑关系类型(4种)和难度(3种)，需要更多实体多样性

- **具体问题**:
  ```
  实体多样性分析：
  - 城市实体：134个（34+100）
  - 每个城市平均出现次数：8000/134 ≈ 60次
  - 如果每次随机选2个城市，期望：8000 × 2 / 134 ≈ 120次/城市

  问题：
  - 热门城市（北京、上海）出现频率可能远高于平均值
  - 小城市可能很少出现
  - 导致模型对热门城市过拟合
  ```

- **影响分析**:
  - **数据偏差**：某些地理区域过度代表，其他区域代表不足
  - **泛化能力差**：模型在训练时未见的实体上表现可能很差
  - **评估偏差**：GeoSR-Bench如果包含训练时少见的实体，准确率会偏低

- **修改建议**:
  1. **扩展地理实体库**：
     - 目标：至少500个地理实体
     - 包括：300个地级市、50个县、50个景点、50个河流/山脉、50个其他
     - 确保地域分布均衡（每个省份至少10个实体）

  2. **设计均衡采样策略**：
     ```python
     def balanced_entity_sampling(num_samples=8000):
         # 确保每个实体至少出现min_freq次
         min_freq = 10
         # 确保每个实体最多出现max_freq次
         max_freq = 30

         # 第一轮：每个实体至少min_freq次
         samples = []
         for entity in entities:
             samples.extend(generate_samples(entity, min_freq))

         # 第二轮：随机补充到目标数量
         while len(samples) < num_samples:
             e1, e2 = random_sample(entities, 2)
             if count_frequency(e1) < max_freq and count_frequency(e2) < max_freq:
                 samples.append(generate_sample(e1, e2))

         return samples
     ```

  3. **添加数据分布报告**：
     - 生成后统计每个实体的出现频率
     - 生成地域分布热力图
     - 确保方差在一定范围内（如CV < 0.3）

  4. **在GeoSR-Bench中包含罕见实体**：
     - 30%的测试数据使用训练时出现频率<15次的实体
     - 测试模型的泛化能力

- **优先级**: P1（建议在数据生成前扩展实体库）

### 2.3 轻微级别问题（可选优化）

#### 🟢 新问题6：缺少数据生成过程中的偏差检测机制

- **位置**: 4.6节数据生成与质量控制
- **问题描述**:
  - 质量控制关注单条数据的质量
  - 但未检测整个数据集的系统性偏差
  - 例如：教师模型可能倾向于生成某种类型的问题

- **具体偏差类型**:
  1. **关系类型偏差**：
     - 教师模型可能更擅长生成方向关系问题
     - 导致数据集中方向关系比例高于设计值（30%）
  2. **地理区域偏差**：
     - 东部城市实体更丰富，导致东部相关问题更多
  3. **答案分布偏差**：
     - 方向问题中，"南方"答案可能多于"北方"
  4. **难度分布偏差**：
     - 生成Hard问题的失败率更高，导致实际Hard比例低于20%

- **修改建议**:
  1. **添加数据分布统计模块**：
     ```python
     def analyze_dataset_bias(dataset):
         report = {
             'relation_type_dist': Counter(d['spatial_relation_type'] for d in dataset),
             'difficulty_dist': Counter(d['difficulty'] for d in dataset),
             'entity_freq': Counter(),
             'region_dist': Counter(),  # 按省份统计
             'answer_dist': {}  # 按答案类型统计
         }
         # 检测偏差
         report['bias_detected'] = {
             'relation_type': entropy(report['relation_type_dist']) < threshold,
             'difficulty': entropy(report['difficulty_dist']) < threshold,
         }
         return report
     ```

  2. **设定偏差阈值**：
     - 关系类型分布：每种类型占比25%±5%
     - 难度分布：Easy 30%±5%, Medium 50%±5%, Hard 20%±5%
     - 实体频率：CV < 0.4（变异系数）
     - 超出阈值的数据重新生成

  3. **在4.6节添加"偏差检测与校正"子节**

- **优先级**: P2（建议添加，提升数据质量）

---

#### 🟢 新问题7：合成数据与原始数据的"等价性"未经验证

- **位置**: C4合成数据蒸馏、Exp6设计
- **问题描述**:
  - C4假设"教师生成的合成数据"与"原始数据"等价
  - 但未验证合成数据在难度、分布、质量上是否与原始数据一致
  - 如果合成数据更简单，Exp6的性能提升可能来自"更简单的数据"，而非C4组件

- **验证需求**:
  ```
  等价性验证清单：
  1. 难度分布：合成数据的Easy/Medium/Hard比例是否与原始数据一致？
  2. 关系类型分布：四种关系类型的比例是否一致？
  3. 答案多样性：合成数据的答案熵是否不低于原始数据？
  4. 推理链质量：合成数据的推理链平均长度是否相当？
  5. 教师模型表现：教师在合成数据和原始数据上的准确率是否相当？
  ```

- **修改建议**:
  1. **添加等价性验证实验**：
     ```
     验证实验设计：
     - 在原始数据上评估教师模型准确率：Acc_orig
     - 在合成数据上评估教师模型准确率：Acc_synth
     - 要求：|Acc_orig - Acc_synth| < 3%

     - 计算原始数据的难度分布熵：H_orig
     - 计算合成数据的难度分布熵：H_synth
     - 要求：|H_orig - H_synth| / H_orig < 10%
     ```

  2. **在C4组件描述中添加"数据等价性保证"**：
     - 说明如何确保合成数据与原始数据等价
     - 或说明合成数据的特性（如"合成数据更困难/更简单"）

  3. **考虑将Exp6重新定位**：
     - 不作为消融实验，而作为"数据质量影响研究"
     - 探讨合成数据vs原始数据的优劣

- **优先级**: P2（如果保留Exp6，则必须验证）
