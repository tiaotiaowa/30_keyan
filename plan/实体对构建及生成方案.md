# GeoKD-SR 双层数据架构设计方案

## Context

### 问题背景
当前实验设计面临一个根本性质疑：空间关系可以由PostGIS精确计算，为什么要让LLM来做？如果过度结构化，模型可能退化为"空间关系计算器"。

### 解决方案
重新定义研究价值——不是让LLM替代PostGIS做计算，而是验证小模型能否从大模型中**蒸馏学习空间推理能力**：
- **推理能力**：根据已知信息推导空间关系结论
- **自然语言理解**：理解不同表述方式的空间问题并准确回答

### 已确认的关键决策
1. 能力定位：推理能力 + 自然语言理解（非知识检索）
2. 评测场景：有坐标/无坐标两种，分开报告
3. 正负例：方向/距离全正例，拓扑/复合约30%负例
4. 数据规模：10,000实体对 × 4种题型 = 最多40,000条，训练时选择性使用
5. 数据架构：双层结构（实体对事实层 + 题目实例层）

## 一、实体对事实层（Layer 1）

### 1.1 结构设计

每个实体对只包含`target_relation`相关的空间事实，按关系类型分5种子结构：

**方向关系对** (`target_relation: "directional"`)
```json
{
  "pair_id": "dir_00142",
  "target_relation": "directional",
  "entity_a": {"name_zh": "武汉市", "centroid": [114.31, 30.59], "type": "city"},
  "entity_b": {"name_zh": "长沙市", "centroid": [112.94, 28.23], "type": "city"},
  "spatial_facts": {
    "direction_8": "东北",
    "azimuth_deg": 29.7
  }
}
```

**距离关系对** (`target_relation: "metric"`)
```json
{
  "pair_id": "metric_00142",
  "target_relation": "metric",
  "entity_a": {"name_zh": "武汉市", "centroid": [114.31, 30.59], "type": "city"},
  "entity_b": {"name_zh": "长沙市", "centroid": [112.94, 28.23], "type": "city"},
  "spatial_facts": {
    "distance_km": 296
  }
}
```

**拓扑关系对** (`target_relation: "topological.contains"` 等5种)
```json
{
  "pair_id": "topo_contains_00001",
  "target_relation": "topological.contains",
  "is_negative": false,
  "entity_a": {"name_zh": "湖北省", "centroid": [112.30, 30.97], "type": "province"},
  "entity_b": {"name_zh": "武汉市", "centroid": [114.31, 30.59], "type": "city"},
  "spatial_facts": {
    "contains": true
  }
}
```

**拓扑负例**
```json
{
  "pair_id": "topo_crosses_00450",
  "target_relation": "topological.crosses",
  "is_negative": true,
  "entity_a": {"name_zh": "长江", "centroid": [113.20, 30.38], "type": "river"},
  "entity_b": {"name_zh": "北京市", "centroid": [116.41, 39.90], "type": "province"},
  "spatial_facts": {
    "crosses": false
  }
}
```

**复合关系对** (direction+topology等)
```json
{
  "pair_id": "comp_dir_topo_00001",
  "target_relation": "composite.direction_topology",
  "is_negative": false,
  "entity_a": {"name_zh": "黄山", "centroid": [118.17, 30.13], "type": "peak"},
  "entity_b": {"name_zh": "安徽省", "centroid": [117.28, 31.86], "type": "province"},
  "spatial_facts": {
    "direction_8": "南",
    "azimuth_deg": 188.5,
    "distance_km": 112,
    "within": true
  }
}
```

### 1.2 数据生成策略：正负例分阶段独立生成

**核心原则**：先全部生成正例（10,000条），再独立生成负例（1,237条），分两个Phase存储，用户按需选择使用。

**正例定义**：空间关系为真的实体对（如"湖北省包含武汉市"→contains=true、"长江穿越湖北省"→crosses=true）
**负例定义**：空间关系为假的实体对（如"长江穿越北京市"→crosses=false、"武汉在河南省内"→within=false）

### 1.5 总数据规模

| Phase | 数量 | 用途 |
|-------|------|------|
| Phase 1 (正例) | 10,000 | 基础训练数据 |
| Phase 2 (负例) | 1,237 | 对比实验（可选替换） |
| **总计** | **11,237** | |

### 1.6 负例使用策略

负例独立存储为`pairs_negative.jsonl`，用户在训练时：
- **方案A**: 仅用Phase 1的10,000正例 → 作为baseline
- **方案B**: 从Phase 1中随机抽取1,237条正例替换为Phase 2负例 → 8,763正 + 1,237负 = 10,000
- **方案C**: Phase 1全部 + Phase 2全部 → 11,237条混合训练
- 对比A vs B vs C的评测结果 → 验证负例对模型空间推理能力的影响

### 1B.5 过采样→筛选流程

```
Step 1: 批量生成（~14,916条正例 + ~1,856条负例）
  ├── 每种关系类型按上表分配过采样
  ├── 存储为 pairs_positive_raw.jsonl / pairs_negative_raw.jsonl

Step 2: 多样性筛选（14,916 → 10,000）
  ├── 实体类型组合均衡：每种组合内按比例筛选
  ├── 方向分布均衡（8方向均匀，TVD < 0.05）
  ├── 距离分布均衡（5区间均匀，TVD < 0.05）
  ├── 实体出现频次控制（同一实体最多出现在N对中）
  └── 难度分布初步校验（Easy:Medium:Hard ≈ 3:5:2）

Step 3: 实体对互斥校验
  ├── 同一对实体（如湖北-武汉）的所有关系类型在同一个split
  ├── 构建实体对→关系类型映射
  └── 冲突解决：将同一对的所有关系归入同一pair_group
```

### 1B.6 实体对互斥与数据划分

**实体对互斥规则**:
- 定义实体对为无序集合 `{entity_a_id, entity_b_id}`
- 同一实体对可以出现在不同关系类型中（如"武汉-长沙"可以同时有directional和metric对）
- 但同一实体对的所有关系类型实例必须在同一个split中

**划分策略**:

| 维度 | 分层键 |
|------|--------|
| L1 | target_relation (11种子类型) |
| L2 | difficulty (3级) |
| L3 | entity_pair_group (实体对互斥) |

**划分流程**:
1. 将所有实体对按pair_group分组
2. 在每个(relation_type, difficulty)层内，按pair_group shuffle
3. 按pair_group级别8:1:1划分（确保同一group的所有关系在同一split）
4. 验证各split的relation_type × difficulty分布TVD < 0.05

**各子类型划分**:

| 子类型 | 总条数 | train(80%) | dev(10%) | test(10%) |
|--------|--------|-----------|---------|----------|
| directional | 2,500 | 2,000 | 250 | 250 |
| metric | 2,500 | 2,000 | 250 | 250 |
| contains | 500 | 400 | 50 | 50 |
| within | 500 | 400 | 50 | 50 |
| touches | 500 | 400 | 50 | 50 |
| crosses | 500 | 400 | 50 | 50 |
| disjoint | 500 | 400 | 50 | 50 |
| C1(方向+距离) | 875 | 700 | 87 | 88 |
| C2(方向+拓扑) | 500 | 400 | 50 | 50 |
| C3(距离+拓扑) | 500 | 400 | 50 | 50 |
| C4(三重) | 625 | 500 | 62 | 63 |
| **合计** | **10,000** | **8,000** | **1,000** | **1,000** |

### 1B.7 重复检测与质量控制

#### 1B.7.1 同类型内去重（必须）

**规则**: 同一关系类型内，不允许出现相同的实体对。

| 关系类型 | 唯一性键 | 示例 |
|---------|---------|------|
| directional | `{entity_a_id, entity_b_id}` 无序 | (武汉,长沙)和(长沙,武汉)视为同一对 |
| metric | `{entity_a_id, entity_b_id}` 无序 | 同上 |
| contains | `(entity_a_id, entity_b_id)` 有序 | (湖北省,武汉市) ≠ (武汉市,湖北省) |
| within | `(entity_a_id, entity_b_id)` 有序 | (武汉市,湖北省) ≠ (湖北省,武汉市) |
| touches | `{entity_a_id, entity_b_id}` 无序 | (河北,山东)和(山东,河北)同一对 |
| crosses | `(entity_a_id, entity_b_id)` 有序 | (长江,湖北省) ≠ (湖北省,长江) |
| disjoint | `{entity_a_id, entity_b_id}` 无序 | 同touches |
| C1-C4 | 按主关系类型的规则 | C2/C3有序, C1无序 |

**实现**: 每种类型采样时使用Python set/frozenset去重。

#### 1B.7.2 实体出现频次控制

**问题**: 如果湖北省出现在500条contains对中，模型可能"记住"湖北省而非学习推理模式。

**规则**:
- **Polygon实体**（province/city/lake）: 同一实体在同一关系类型中**最多出现30次**
- **Point实体**（peak/attraction等）: 同一实体在同一关系类型中**最多出现20次**
- **Line实体**（river/road/railway）: 同一实体在同一关系类型中**最多出现40次**（line实体较少）

**对contains的具体影响**:
- province(34) × 30次 = 最多1,020条 → 需要750条contains → ✅ 可行
- city(476) × 20次 = 最多9,520条 → 需要750条 → ✅ 可行
- 但province→city只有474对，34个province平均14次 → ✅ 安全

#### 1B.7.3 跨类型实体对协调

**核心问题**: 同一实体对（如湖北省-武汉市）可以出现在多个关系类型中（contains + directional + metric + C2 + C3 + C4），但所有这些数据必须在同一个split中。

**协调策略**:
1. **Phase 1**: 每种关系类型独立采样（允许跨类型重叠）
2. **采样后**: 构建全局实体对→关系类型映射表
3. **划分时**: 按pair_group（无序实体对集合）统一分配split

```python
# 构建pair_group
pair_groups = defaultdict(set)  # frozenset({a_id, b_id}) -> {relation_type1, relation_type2, ...}
for pair in all_pairs:
    key = frozenset([pair['entity_a_id'], pair['entity_b_id']])
    pair_groups[key].add(pair['target_relation'])
```

4. **泄露分析标记**:
   - **seen_pair**: test中实体对的两个实体分别在train的不同对中出现过
   - **unseen_pair**: test中至少一个实体在train中从未出现
   - 分别计算准确率，差异<5%为佳

#### 1B.7.4 紧张分配的实体对管理

以下组合可用对数接近过采样需求，需特别注意：

| 组合 | 可用 | 过采样 | 余量 | 管理 |
|------|------|--------|------|------|
| river→province crosses | 102 | 101 | **1** | 降为100，差额补入road→province |
| province→province touches | 62 | 62 | **0** | 全量使用，不设余量 |
| province→airport contains | 60 | 40 | 20 | 安全 |
| city→lake contains | 59 | 40 | 19 | 安全 |

**river→province crosses修正**:

| 编号 | 实体A(线) | 实体B(面) | 修正后过采样 | 最终目标 | 可用 |
|------|---------|---------|------------|---------|------|
| TX1 | river | province | 100 | 67 | 102 ✅ |
| TX2 | river | city | 180 | 120 | 366 ✅ |
| TX3 | road | province | 130 | 87 | 239 ✅ |
| TX4 | road | city | 170 | 113 | 831 ✅ |
| TX5 | railway | province | 90 | 60 | 135 ✅ |
| TX6 | railway | city | 80 | 53 | 442 ✅ |
| **合计** | | | **750** | **500** | |

#### 1B.7.5 包含策略选择规则

| A类型 | B类型 | 策略 | 原因 |
|-------|-------|------|------|
| province(Polygon) | city(Polygon) | **质心包含**: province.contains(city.centroid) | 行政边界不完全嵌套，质心更稳健 |
| province(Polygon) | lake(Polygon) | **质心包含**: province.contains(lake.centroid) | 湖泊可能跨越省界 |
| city(Polygon) | lake(Polygon) | **严格包含**: city.contains(lake) | 城市内湖泊应严格在内 |
| Polygon | Point(POI) | **点包含**: polygon.contains(point) | Point几何精确 |
| Polygon | Polygon(嵌套) | **严格包含**: polygon_a.contains(polygon_b) | 小Polygon应严格在大Polygon内 |

| 约束 | 说明 | 验证方式 |
|------|------|---------|

### 6.3 Step 1a: Phase 1 正例实体对生成

**方向关系 (directional) — 2,500对**:
```sql
-- 从所有实体对中随机采样，计算8方位方向
SELECT a.entity_id, b.entity_id,
       ST_Azimuth(ST_Transform(a.centroid,3857), ST_Transform(b.centroid,3857)) as azimuth
FROM geokr_entity a, geokr_entity b
WHERE a.entity_id < b.entity_id  -- 避免重复
ORDER BY RANDOM() LIMIT 2500;
```
- azimuth→8方位映射: N(337.5-22.5), NE(22.5-67.5), E(67.5-112.5), ...
- 确保entity_type组合多样性：city-city, city-peak, province-city, peak-lake等

**距离关系 (metric) — 2,500对**:
```sql
SELECT a.entity_id, b.entity_id,
       ST_DistanceSphere(a.centroid, b.centroid) as distance_km
FROM geokr_entity a, geokr_entity b
WHERE a.entity_id < b.entity_id
ORDER BY RANDOM() LIMIT 2500;
```
- 距离分布控制：< 100km(500对), 100-500km(500对), 500-1000km(500对), 1000-2000km(500对), 2000+km(500对)

**拓扑包含 (topological.contains) — 500对**:
```sql
SELECT a.entity_id, b.entity_id, true as contains
FROM geokr_entity a, geokr_entity b
WHERE a.type IN ('province', 'city')  -- 容器实体
  AND ST_Contains(a.geom, b.centroid)  -- A的几何包含B的质心
ORDER BY RANDOM() LIMIT 500;
```
- 实体组合: province×city(~474), province×POI(~598), city×POI(~599)
- 采样时保持组合多样性

**拓扑包含于 (topological.within) — 500对**:
- 同contains但视角反转(A是POI，B是province/city)

**拓扑相邻 (topological.touches) — 500对**:
```sql
-- Province-Province相邻
SELECT a.entity_id, b.entity_id
FROM geokr_entity a, geokr_entity b
WHERE a.type='province' AND b.type='province'
  AND a.entity_id < b.entity_id
  AND ST_Touches(a.geom, b.geom);
-- City-City相邻
SELECT a.entity_id, b.entity_id
FROM geokr_entity a, geokr_entity b
WHERE a.type='city' AND b.type='city'
  AND a.entity_id < b.entity_id
  AND ST_Touches(a.geom, b.geom);
```

**拓扑穿越 (topological.crosses) — 500对**:
```sql
SELECT a.entity_id, b.entity_id
FROM geokr_entity a, geokr_entity b
WHERE a.geometry_type='Line' AND b.geometry_type='Polygon'
  AND ST_Crosses(a.geom, b.geom)
ORDER BY RANDOM() LIMIT 500;
```
- 来源: river×province(~186), river×city(~更多), road×province, railway×province

**拓扑分离 (topological.disjoint) — 500对**:
```sql
SELECT a.entity_id, b.entity_id
FROM geokr_entity a, geokr_entity b
WHERE ST_Disjoint(a.geom, b.geom)
  AND a.entity_id < b.entity_id
ORDER BY RANDOM() LIMIT 500;
```

**复合关系 — 2,500对**:
- 从已有的directional和metric对中，附加计算拓扑关系
- C1(direction+distance): 875对 — 随机采样实体对，计算方向+距离
- C2(direction+topology): 500对 — 采样实体对，计算方向+拓扑
- C3(distance+topology): 500对 — 采样实体对，计算距离+拓扑
- C4(三重): 625对 — 采样实体对，计算方向+距离+拓扑
