```json
    "difficulty": "easy"
  },
  "choice": {
    "question": "...\nA. ...\nB. ...\nC. ...\nD. ...",
    "options": {"A": "...", "B": "...", "C": "...", "D": "..."},
    "answer": "B",
    "answer_structured": {"type": "option", "value": "B", "correct_answer": "..."},
    "difficulty": "medium"
  },
  "fill_blank": {
    "question": "...______...",
    "answer": "东北",
    "answer_structured": {"type": "keyword", "value": "东北"},
    "difficulty": "easy"
  },
  "open_qa": {
    "question": "...",
    "answer": "...",
    "answer_structured": {"type": "free_text", "direction": "东北", ...},
    "difficulty": "medium"
  }
}
```

### 4.4 核心约束（5条，所有模板共享）

1. **不做空间计算**: 所有事实由PostGIS精确计算并直接给出
2. **直接派生**: answer_structured必须从spatial_facts直接派生，不得修改数值
3. **表达一致**: answer的自然语言表达必须与answer_structured完全一致
4. **风格各异**: 4道题目的提问风格必须各不相同
5. **严格JSON**: 输出严格为JSON，不添加额外文本

## 八、验证与质量控制

### 8.1 双层验证机制

**第一层: JSON结构验证**
- 检查 `true_false` / `choice` / `fill_blank` / `open_qa` 四个字段存在
- 检查每个题型包含 `question` 和 `answer_structured`
- 检查选择题包含 `options`（至少2个选项）

**第二层: 空间事实一致性验证**
- **方向一致性**: answer_structured.direction == spatial_facts.direction_8（精确匹配）
- **距离一致性**: |answer_structured.distance_km - spatial_facts.distance_km| ≤ 1（允许1km误差）
- **拓扑一致性**: answer_structured.contains/within/... == spatial_facts.contains/within/...（布尔匹配）

### 8.2 失败处理策略

| 失败类型 | 最大重试 | 重试操作 |
|---------|---------|---------|
| JSON解析失败 | 1次 | 追加"请输出纯JSON"提醒 |
| Schema验证失败 | 1次 | 追加"必须包含4种题型"提醒 |
| 两次均失败 | — | 记录error.log，跳过该条 |

## 附录A: 常见问题

### Q1: 为什么用DeepSeek-V3.2而不用Qwen？

当前小样本测试基于DeepSeek-V3.2通过DashScope调用。使用指导.md中的Pipeline代码基于Qwen系列编写。两者API兼容（均为DashScope OpenAI兼容模式），可按需切换。建议：
- **质量优先**: qwen-max 或 deepseek-v3.2
- **成本优先**: qwen-plus
- **速度优先**: qwen-turbo

### Q2: API Key配额与费用估算

```
单条Token消耗: ~3,275-4,251 tokens
总Token估算: 11,237条 × 3,800 tokens/条 ≈ 42.7M tokens
费用估算: 参考阿里云百炼实时定价
```

### Q3: 如何处理负例模板中的 actual_container？

数据中不含actual_container字段，模板中该占位符设为空字符串。LLM在生成负例题目时会根据地理常识自动推断actual_container（如"武汉市不属于河南省→actual_container=湖北省"）。

### Q4: 数据中的 entity 没有 geometry_type 字段怎么办？

模板中 `{{entity_a_geometry_type}}` 和 `{{entity_b_geometry_type}}` 占位符设为空字符串。LLM可根据entity type自行推断（province/city/lake→Polygon, peak/attraction/...→Point, river/road/railway→Line）。

### Q5: 方向/距离/复合关系C1为什么没有负例模板？

方向和距离总有确定值（不存在"方向为假"或"距离为假"的概念），因此不设计负例。只有拓扑关系和含拓扑的复合关系才存在真/假之分。
